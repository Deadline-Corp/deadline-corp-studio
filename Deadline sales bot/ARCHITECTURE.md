# DEADLINE Sales Bot — архитектура (карта проекта)

Документ, чтобы быстро ориентироваться: что где лежит, как течёт запрос, как
откатывать и править. Обновлён 2026-06-03.

---

## 1. Что это
AI-бот квалификации лидов для веб-студии. Лид пишет (сайт-виджет / Telegram) →
бот отвечает (LLM) → собирает контакты → ведёт по воронке до созвона → зеркалит
всё в HubSpot CRM. Менеджер видит карточку и ведёт сделку дальше.

## 2. Стек
- **FastAPI** (async) + **uvicorn** — HTTP-слой.
- **LLM:** `meta-llama/llama-3.3-70b-instruct` через **OpenRouter** (можно сменить
  на Grok/др. одной строкой — см. `_LLM_PRIMARY_MODEL`).
- **Postgres** (+ pgvector для RAG) — **источник правды**. SQLAlchemy (sync, psycopg2).
- **HubSpot** — зеркало (CRM), через адаптер + очередь событий.
- **Railway** — хостинг (Docker). Деплой `railway up --detach`.

## 3. Главное правило архитектуры
**Postgres = источник правды. HubSpot = зеркало.** Бот пишет в Postgres синхронно,
а в HubSpot — асинхронно через durable-очередь с ретраями. Если HubSpot лёг —
бот работает, события копятся и до-отправляются. **Никогда не блокируем ответ
лиду на CRM-вызове.**

**Хард-правило по llama:** модель НЕНАДЁЖНО следует текстовым инструкциям. Поэтому
все жёсткие гарантии (захват имени/email/канала, бронь, капитализация, анти-спам
слотами) держим **в КОДЕ**, а не в промпте. Промпт — только тон и общая логика.

## 5. Поток одного сообщения (`_handle_message` в main.py)
```
1. Deep-link: "/start sess_xxx" из телеги → склейка с сайт-кастомером (identity.py)
2. Identity: (канал, external_id, email?) → Customer (создать/найти)  [identity.py]
3. Conversation: get_or_create по треду канала
4. Имя: extract_lead_name / голый ответ на "как вас зовут?"  [reply_polish.py]
5. История: последние 30 сообщений (+ recall прошлых диалогов)
6. Сигналы: lead_score / temperature / interaction_type  [lead_signals]
7. Бронь созвона: парс времени/канала, анти-дабл-бронь  [scheduling.py]
8. RAG + промпт + LLM-вызов (OpenRouter)
9. Пост-гарды на ответ (reply_polish): капитализация, анти-spam, и т.д.
10. Handoff-проверка: есть контакт+бриф → квалифицируем, шлём бриф оператору
11. CRM-диспатч (НЕ блокирует): контакт/сделка/задачи/лог-сообщения в очередь
12. Commit транзакции, ответ лиду
```

## 6. CRM-синхронизация (что попадает в HubSpot)
Через очередь `crm_events` → воркер → `HubSpotAdapter`:
- **Контакт:** имя, email, телефон, telegram_handle, канал, скор, температура, тип лида.
- **Сделка:** название (суть запроса), стадия воронки, бриф, тип проекта, время созвона.
- **Задачи:** «подхватить», «подтвердить созвон −1д», «через час −1ч», «связаться позже».
- **Вся переписка:** каждое сообщение лида и бота → Note на контакте (таймлайн).
  → **диалог виден прямо в карточке HubSpot.**
- **Дедуп:** один контакт на человека (email/phone/tg), одна открытая сделка;
  кросс-канальная склейка через HubSpot Merge.

## 7. Воронка (8 стадий)
🆕 Новый лид → 💬 В диалоге → ✅ Квалифицирован → 📞 Созвон назначен → 📄 КП →
💰 Аванс → ✅ Сдано · ❌ Проигран.
**Бот авто-двигает до «Созвон назначен»**, дальше — менеджер вручную.
Стадии заданы в `hubspot.py::STAGE_DEFS`; `_reconcile_stages` только ДОБАВЛЯЕТ
недостающие (никогда не удаляет) — удаление стадий только вручную в HubSpot.

## 8. Память
- Активный диалог: последние **30 сообщений** в контекст LLM (`main.py` limit=30).
- Кросс-сессия: `get_recent_messages_with_recall` подтягивает хвост прошлых
  диалогов того же кастомера (вернувшийся лид узнаётся).
- Долго: вся переписка в Postgres (`messages`) + факты в `customers.profile_data`.
- Для ОЧЕНЬ длинных циклов наилучшее — рулонный summary (TODO, не реализован).

## 9. Отложенные действия / напоминания (`scheduled_actions`)
- `followup_message` (executor=bot): «напишите позже» → бот сам пишет в срок (крон).
- `call_booked` + `call_reminder` (−3ч/−1ч лиду и админу).
- Бот-нудж молчуну: вовлечённый лид пропал ≥1ч → бот один раз пингует (крон, `cron.py`).
- Крон гоняет всё это: `run_due_followups`, `run_due_call_reminders`, `sweep_once`.
  Ручной триггер: `POST /admin/cron/sweep` (Bearer TRAINING_AUTH_TOKEN).

## 11. Известные риски / эксплуатация
- 🔴 **idle-freeze:** контейнер может зависнуть в простое (был 2026-06-03: 4ч тишины
  в логах, помог рестарт). Митигации: `_heartbeat_loop` (пульс+диагностика),
  `railway.json` healthcheck, **внешний монитор** (UptimeRobot на `/health` каждые
  5 мин — обязательно перед трафиком).
- 🟡 Холодный старт ~30-60с после простоя (виджет показывает «обрабатываю…»).
- 🟡 llama иногда повторяется / может сказать «записал», когда код не забронировал
  (текст модели разошёлся со стейтом). Гарды ловят часть.
