## 1. ВИДЕНИЕ (куда идём — цель проекта)

Мы строим **умную систему управления продажами на базе агентов** — собственную
гибкую CRM/Sales-OS, которой можно управлять через визуальный интерфейс и которая
настраивается практически под любой бизнес.

Суть в одном абзаце: система **понимает кто есть кто** (клиент / потенциальный
клиент / сотрудник / руководитель), **ведёт клиентов по воронкам**, **ведёт
расписание и созвоны**, **сама решает простые задачи и передаёт менеджеру
сложные**, **отсекает спам и понимает контекст** (шутят/не шутят, реальный
интерес или «написал-пропал»), **напоминает** и клиентам, и сотрудникам, и сама
**реактивирует спящих**. Всё это — из одного окна, интуитивно, без кода.

Ключевые принципы:
- **Гибкость и подключаемость.** Своя воронка/канбан/календарь по умолчанию, но при
  необходимости можно подключить внешние сервисы (внешний календарь, сторонняя CRM,
  новые каналы трафика). Подключение — самостоятельное, через интерфейс, без
  программиста.
- **Self-service каналы.** Пользователь сам подключает Telegram / WhatsApp /
  Instagram / Facebook через приложение (вставил ключи по инструкции → канал ожил).
- **Умный мозг.** Агент(ы) с реальным пониманием контекста, а не скрипт. Жёсткие
  гарантии — в коде, «понимание» — в модели.
- **Под любую нишу.** Продажи везде похожи; ниша задаёт воронку, тон, прайс, скрипты.
- **Видно как работает.** Интерфейс показывает живую картину: каналы, воронку,
  переписки в одном месте, задачи, аналитику.

Бизнес-модель: сначала бесплатно отдаём товарищам-предпринимателям (пилот за
фидбэк), позже — платный SaaS. База в Таиланде → 152-ФЗ пока не блокер.

## 2. ДВА СЛОЯ СИСТЕМЫ

### Слой 1 — Бот (агент первого контакта). РАБОТАЕТ, в проде.
Лид пишет (виджет на сайте / Telegram) → бот отвечает (LLM), даёт цены «от $X»,
собирает контакт + суть задачи, доводит до **созвона с человеком**, греет молчунов,
ставит задачи, зеркалит всё в HubSpot. Путь клиента:
`Переписка → LEAD → Квалификация → Созвон → ТЗ → КП → Предоплата → Исполнение → Закрытие`.

### Слой 2 — Панель управления (`/admin/ui/`). РАБОТАЕТ, 12 страниц.
React/Vite SPA под FastAPI. Страницы: **Canvas** (связи), **Funnel** (воронка-канбан),
**Inbox** (переписки всех каналов), **Brain** (мозг/правила бота), **Automations**
(глобальные правила), **Channels** (подключение каналов), **Calendar**, **Tasks**,
**Analytics**, **Settings**, **Onboarding** (визард + spotlight-обучение), **Login**.

## 3. ТЕКУЩАЯ АРХИТЕКТУРА (коротко; детали — ARCHITECTURE.md)

- **FastAPI** (async) + **uvicorn**. **Postgres (+pgvector)** = источник правды.
  **HubSpot** = зеркало через durable-очередь `crm_events` с ретраями.
- **Мозг:** OpenRouter, `meta-llama/llama-3.3-70b-instruct` (primary) + DeepSeek
  (fallback). Конфиг в `main.py` (`openrouter_model`), переключается env-переменной
  `OPENROUTER_MODEL` без правок кода. Доступные провайдеры по ключам в env:
  **OpenRouter, Groq, Ollama**. **Gemini/Google-ключа НЕТ** (ни локально, ни на проде).
- **Хард-правило:** llama ненадёжно слушает текстовые инструкции → все жёсткие
  гарантии (захват контакта, бронь, капитализация, анти-спам) держим **в КОДЕ**
  (`services/reply_polish.py`, `scheduling.py`, `main.py` пост-гарды), не в промпте.
- **Сервисы** (`services/`): identity, funnel, funnel_store, scheduling, crm_dispatch,
  crm_queue, scheduled_actions, cron, triggers (7 проверок эскалации), warming,
  scoring, temperature, lead_signals, escalation, returning_lead, channel_prefs,
  tenant (мультиарендность), prompt_store, bot_settings, training, automation.
- **Деплой:** Railway, `railway up --detach` из папки сервиса. Виджет — отдельно
  в репо сайта (`widget.js`, GitHub Pages). Ветка: `feature/call-booking`.

## 6. ДЕБАТЫ И РЕШЕНИЯ (по спорным развилкам)

### 6.1. Умнее мозг — какую модель?
**Проблема:** llama-3.3-70b иногда уводит не по делу. Gemini-ключа в системе НЕТ
(вопреки ожиданию — проверено, его никто не добавлял).
**Варианты:** (A) добавить Gemini 2.x Flash/Pro — нужен бесплатный `GOOGLE_API_KEY`;
(B) переключить OpenRouter на более сильную модель (напр. свежий Qwen/Claude Haiku/
DeepSeek-V3) — 1 env-переменная, обратимо, но дороже; (C) Groq (ключ ЕСТЬ) — очень
быстро, но набор моделей ограничен.
**РЕШЕНИЕ:** сделать **провайдер-абстракцию мозга** (OpenRouter | Gemini | Groq —
переключатель в Настройках/env), чтобы модель менялась без правок кода. Дефолт пока
OpenRouter; как только пользователь даст `GOOGLE_API_KEY` — включаем Gemini Flash
(дёшево+умно). Это снимает спор «какая модель» — система станет модель-агностичной.
→ **ВОПРОС 1 пользователю.**

### 6.2. Фрагментация карточек — «умная гибкая» модель вместо тупого persistent-id
**Проблема:** на сайте без email каждый визит = новая карточка. Старое решение
(жёсткий cookie-id) — плоско.
**РЕШЕНИЕ (умная модель, как просил пользователь):**
- **Черновик-карточки (draft).** Любой первый контакт = лёгкая draft-карточка
  (не «настоящая» сделка). Без шума в воронке.
- **Промоушен по сигналам.** Draft → реальная карточка только когда лид «реально
  вошёл»: дал контакт ИЛИ ≥N осмысленных сообщений ИЛИ выраженный интерес/бронь.
- **Авто-склейка.** Появился email/телефон/tg → ищем существующего клиента, мёрджим
  draft в него (без дублей).
- **Авто-уборка мусора.** Draft без признаков жизни (1 сообщение, пропал/заблокировал,
  спам) → через TTL помечается dead и не висит в CRM. Удаление — мягкое (архив),
  по правилу never-delete.
- **Cookie/localStorage id** виджета — как технический «клей» внутри draft-логики,
  чтобы в рамках визита/возврата не плодить draft’ы.
Реализация: `identity.py` + новый слой `lead_lifecycle`/расширение `lead_signals`.

### 6.3. Календарь — свой + внешний
**РЕШЕНИЕ:** свой календарь — источник правды для слотов/созвонов/дедлайнов (уже
есть страница Calendar + бронь в `scheduling.py`). Внешний (Google Calendar) —
**опциональная двусторонняя синхронизация** через OAuth, подключается в Настройках.
ICS-экспорт (read-only подписка) — лёгкий первый шаг без OAuth. №15 = довести свой
календарь (управление слотами/рабочими окнами из UI), внешнее — следом.

### 6.4. Каналы и «не перегружать интерфейс»
**РЕШЕНИЕ:** раздел **Channels уезжает в Настройки** (вкладка «Каналы и
интеграции»). На главном интерфейсе остаётся только индикатор подключённых каналов.
**Self-service connect:** карточка канала с пошаговой инструкцией + поля для ключей
(Telegram-токен / Wazzup-ключ / Meta App) → сохранил → бэкенд поднимает канал.
Telegram уже так работает; WA/IG/FB — каркас готов, оживут с ключами пользователя.
Туда же — внешний календарь и сторонняя CRM.

### 6.5. Реактивация спящих клиентов
Пользователь считает риск бана приемлемым для **разового** запуска диалога.
**РЕШЕНИЕ:** двухуровнево. (1) Безопасно и сразу — реактивация в каналах, где есть
официальный API/наш бот (Telegram, наш виджет, позже WA Business API). (2)
Браузер-агент (протыкать вручную-подобно) — **только как ручной, подтверждаемый
оператором разовый прогон**, с явной кнопкой, лимитами и предупреждением. Не
авто-массово (там бан реален). → **ВОПРОС 5.**

### 6.6. Агентный «мозг» без кода
**РЕШЕНИЕ:** ввести понятие **ролей собеседника** (клиент/сотрудник/руководитель/
спам) + **инструменты агента** (создать задачу, забронировать, сменить стадию,
напомнить, передать менеджеру, отсечь спам). Правила задаются в Brain/Automations
текстом и кнопками. Цель — чтобы владелец без кода описал «если X — делай Y».

## 9. ЛОКНУТЫЕ РЕШЕНИЯ 2026-06-13 (ответы пользователя)

- **Автономность — главное требование:** система отдаётся товарищу «под ключ», он
  сам подключает СВОИ мозги/ключи/агентов по инструкции. Значит: мультипровайдер
  мозга + per-tenant ключи в Настройках + инструкции в UI. Никакой нашей привязки.
- **Мозг:** делаем переключатель провайдера (OpenRouter | Gemini | Groq), модель-
  агностично. OpenRouter остаётся ДЕФОЛТОМ в проде (живой трафик не трогаем), Gemini
  включается явным `LLM_PROVIDER=gemini`. Для теста есть рабочий Google-ключ
  (`AIzaSy…`, лежит в D:\SMM Easy One\.env и D:\Христианский бот\.env).
- **Агентность (решено по ресёрчу):** 2 уровня. (1) Tool-use слой (function-calling,
  ВСЕГДА вкл): действия в нашей CRM, поиск, чтение статусов — детерминированно,
  безопасно. (2) Браузер-агент **browser-use** в отдельном sandbox, **opt-in, по
  запросу/по кнопке оператора**, с лимитами (timeout, бюджет, allowlist доменов),
  без прямого доступа к WhatsApp/CRM. НЕ ставить OpenHands в прод (это dev-инструмент).
  НЕ давать автономную масс-реактивацию (бан).
- **Приоритет сейчас:** работать под DEADLINE (трафик уже идёт) + подключить WhatsApp
  владельца. Универсальность под другие ниши — потом, но архитектурно заложить.
- **WhatsApp:** официальный **Meta WhatsApp Cloud API** (надёжно, платформа бесплатно,
  нет риска бана как у «серого» QR). Wazzup24 QR — только для быстрого демо, не для
  боевого номера. Исходящие/реактивация — только approved-шаблоны + opt-in + 24ч-окно.
  Ключи владельца в Настройках: PHONE_NUMBER_ID, WHATSAPP_TOKEN (permanent system user),
  WABA_ID, WEBHOOK_VERIFY_TOKEN. Узкое место — Business Verification (1–7 дней) → начать сразу.
- **Календарь:** свой = источник правды; внешний — лёгкий коннектор (ICS-подписка
  первым шагом, Google OAuth позже). Не переусложнять.
- **Сторонняя CRM:** своя (HubSpot-зеркало + панель) в приоритете; коннектор к чужим
  (Bitrix24/AmoCRM/HubSpot напрямую) — «на всякий случай», задел в коде уже есть.
- **Сотрудники/роли:** добавить «Команду» (их Telegram/телефоны) → бот понимает, кто
  свой, маршрутизирует уведомления и напоминалки по сотрудникам.

### Безопасность живого бота
Бот в проде принимает РЕАЛЬНЫЕ заявки. Рискованные изменения мозга/ядра НЕ деплоим
впопыхах — только после локального теста, отдельным коммитом, с быстрым откатом.
Аддитивное (новый канал WhatsApp, Настройки) безопаснее — не трогает сайт/Telegram-поток.
