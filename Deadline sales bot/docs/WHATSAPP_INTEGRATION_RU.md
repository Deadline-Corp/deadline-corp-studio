# WhatsApp для DEADLINE — план подключения + безопасный тест (2026-06-14)

Задача (DEADLINE prod, НЕ копия Кирила): подключить WhatsApp к боту так, чтобы он
обрабатывал входящих клиентов — **тянул на созвон, понимал что конкретно нужно,
понимал голосовые, составлял техзадание и отправлял администратору**. Сложность:
**на номер уже пишут живые клиенты**, команда отвечает вручную → тест должен быть
безопасным, нельзя «уронить» живое общение.

---

## 1. Что в коде УЖЕ есть (задеплоено на прод, «спит» пока нет WHATSAPP_TOKEN)
- `GET/POST /webhooks/whatsapp` — верификация + приём (подпись через `WHATSAPP_APP_SECRET`).
- `channels/whatsapp.py`: `parse_whatsapp_webhook` (берёт **только текст**) + `send_whatsapp_reply`
  (free-form ответ внутри 24-часового окна).
- Идёт через тот же полный пайплайн `_handle_message`, что и остальные каналы →
  **сейчас это автоответ клиенту**: бот сам отвечает, тянет на созвон, ведёт воронку.
- Настройки самообслуживания: `WHATSAPP_TOKEN / _PHONE_NUMBER_ID / _VERIFY_TOKEN / _APP_SECRET`
  (вставляются в Настройки → Каналы).

## 2. Чего НЕ хватает под задачу (надо дописать)
- **Голосовые.** Парсер пропускает `type != text`. В WhatsApp голос = `type:audio, voice:true`.
  Нужен путь: вебхук audio → `GET /<media_id>` (URL живёт ~5 мин) → скачать с Bearer →
  ogg/opus → FFmpeg в wav → Whisper/Groq (уже в стеке для Telegram) → дальше как текст.
- **Режим «черновик → администратору».** Сейчас бот отвечает клиенту сразу. При живых
  клиентах безопаснее сначала режим как в Instagram-модуле: бот понимает запрос, готовит
  ответ/ТЗ → шлёт администратору в Telegram кнопками «Отправить / Поправить / Игнор» →
  отправляет только после одобрения. Включается флагом, по умолчанию выключен.
- **Сборка ТЗ → админу.** Новое поведение: из переписки (вкл. расшифрованный голос) бот
  собирает структурированное техзадание (что нужно, объём, сроки, бюджет, контакт) и шлёт
  администратору. Переиспользует AI-копилот + диспетч задач оператору.
- **Echo-фильтр для Coexistence** (см. ниже): сообщения, отправленные командой из приложения,
  приходят в вебхук как `smb_message_echoes` — на них НЕ отвечать автоматически.

## 3. Главная развилка подключения номера, на который УЖЕ пишут

| Путь | Что с ручными ответами | Как подключается | Вывод |
|---|---|---|---|
| **Прямая миграция в Cloud API** | ❌ номер становится **API-only**, приложение отваливается, история не переносится | Embedded Signup, перед этим удалить WhatsApp с телефона | НЕ подходит — теряем ручные ответы |
| **Coexistence (рекоменд.)** | ✅ номер работает **И в приложении, И через API** одновременно | через BSP/Tech-провайдера (Whautomate, ChakraHQ, PickyAssist…), скан QR из WhatsApp Business App | ✅ то, что нужно: бот + люди на одном номере |

**Coexistence** — официальная фича Meta, GA с 6 мая 2025. Требования: WhatsApp **Business
App** 2.24.17+, номер в приложении ≥7 дней (лучше 1-2 мес), Meta Business Portfolio,
подключение через BSP. Бот получает все входящие + «эхо» ручных ответов команды (их
фильтруем). Ограничения: throughput 5-20 msg/sec (нам хватает), отвязываются linked-devices
(WhatsApp Web/десктоп), приложение надо открывать ≥1 раз в ~13 дней.

## 4. ⚠️ Критический риск — блокировка WhatsApp в РФ
По данным research, WhatsApp заблокирован в РФ на уровне провайдеров **с февраля 2026**.
DEADLINE крутит рекламу в т.ч. на РФ-аудиторию (см. [[project_deadline_ru_ads]]). Если
клиенты — из РФ без VPN, WhatsApp для них ненадёжен **независимо от настройки API**.
То, что сейчас пишут, может значить: клиенты не в РФ / с VPN / блок не везде жёсткий.
**Это надо прояснить до вложений в подключение** — иначе можно настроить канал, которым
половина аудитории не сможет пользоваться. Альтернатива для РФ-аудитории — Telegram (уже
подключён) или прямой чат на сайте.

## 5. Деньги (2026, per-message с 1 июля 2025)
- **Service-сообщения** (ответы на входящие в окне 24ч) — **бесплатно, без лимита**. Наш
  основной кейс (клиент пишет первым → бот отвечает) почти весь сюда → ≈$0.
- Платно — только проактивные **шаблоны** вне окна 24ч: Utility/Auth ≈$0.04, Marketing
  ≈$0.08 за сообщение (РФ-тариф). Шаблон одобряется 15-30 мин (иногда до 48ч).
- BSP сверху берёт ~$0.003-0.01/сообщение (зависит от провайдера).

## 6. Безопасный тест — 3 фазы (от нулевого риска к бою)

**Фаза 0 — тестовый номер Meta (риск 0, живых клиентов не трогаем).**
Meta даёт бесплатный тест-номер (90 дней, до 5 получателей-из-команды, реальные клиенты
на него писать не могут). На нём проверяем: верификацию вебхука, приём текста, **полный
путь голосового** (скачивание media + транскрипция), отправку ответа, режим черновик→админ.
Параллельно запускаем Business Verification (2-5 дней).

**Фаза 1 — Coexistence в режиме «тень/черновик» на боевом номере.**
Подключаем реальный номер через Coexistence (приложение продолжает работать, команда
отвечает как раньше). Бот включён в режиме **только черновик администратору** — он читает
входящих, понимает запрос/голос, готовит ТЗ и предложку ответа, но клиенту НЕ пишет сам,
шлёт админу на одобрение. Так проверяем качество на реальном трафике без риска.

**Фаза 2 — постепенный автоответ.**
Когда качество устраивает — включаем автоответ (сначала, например, только в нерабочие часы
или только для типовых вопросов), бригада/админ видит всё в Inbox и в любой момент жмёт
«Взять на себя» (бот замолкает по этому диалогу — `operator_takeover` уже есть).

## 7. Что нужно от Meta (боевой запуск)
Meta Business Manager + Business Verification (2-5 дн, до неё лимит 250 диалогов/день) +
WABA + System User с **бессрочным** токеном + permissions `whatsapp_business_messaging` и
`whatsapp_business_management`. App Review НЕ нужен (свой бизнес, не ISV).

## 8. Открытые решения (нужен ответ владельца)
1. **Где клиенты географически?** (из-за блока РФ — см. §4). Решает, имеет ли смысл WhatsApp.
2. **Команда отвечает на этот номер вручную и должна продолжать?** Да → Coexistence (§3).
3. **Сразу автоответ или сначала черновик→админ?** Рекомендация — черновик→админ (§6 Фаза 1),
   потом автоответ. Особенно учитывая «составлять ТЗ и отправлять администратору» из ТЗ.

## 9. Что ПОСТРОЕНО (2026-06-14, ветка feature/call-booking, аддитивно за флагами)
- ✅ **A) WhatsApp-голосовые.** `channels/whatsapp.py`: `download_whatsapp_media`
  (GET /media-id → url → скачивание с Bearer) + `parse_whatsapp_webhook` теперь async,
  тип `audio` → скачивание ogg/opus → `transcribe_voice` (тот же Groq Whisper, что в
  Telegram; FFmpeg НЕ нужен). Сбои → понятная заглушка клиенту.
- ✅ **B) Режим «черновик → администратору».** Флаг `wa_draft_mode` (bot_settings). ON →
  ответ бота НЕ уходит клиенту, а летит администратору (`manager_chat_id`, иначе
  `TELEGRAM_CHAT_ID`) в Telegram с кнопками **✅ Отправить / 🚫 Не отправлять**. Черновик
  хранится в `conversations.pending_wa_draft` (миграция 016). Кнопки обрабатываются в
  `_handle_operator_callback` (`wad:`/`wadx:`). Переключатель — в UI Каналы → WhatsApp.
- ✅ **C) Сборка ТЗ из переписки.** `_compose_tech_spec` (флаг `wa_techspec`, деф. ON):
  по последним репликам (вкл. расшифрованный голос) LLM собирает структурное ТЗ (что нужно
  / объём / адрес / сроки / бюджет / контакт / что уточнить) — прикладывается к черновику админу.
- ✅ **D) Echo-фильтр Coexistence.** Пропуск поля `smb_message_echoes` + сообщений с нашего
  собственного номера (`display_phone_number`) — бот не отвечает на ручные ответы команды.
- ✅ **E) Одобрение прямо в панели «Переписки» (2026-06-14).** Входящие видны в карточке
  диалога. В режимах наблюдения/черновика бот НЕ публикует ответ как отправленный
  (`_handle_message._wa_route_answer` «удерживает» его, `MessageResponse.suppress_send`) —
  кладёт предложенный ответ в `conversations.pending_wa_draft`. В карточке (`ConversationDrawer`)
  блок **«🤖 Бот предлагает ответить»** с редактируемым текстом + кнопки **✅ Отправить /
  ✏️ (правка) / 🚫 Отклонить** (эндпоинт `POST /conversations/{id}/wa-draft`). Кнопка
  **«🤖 Разрешить боту вести диалог»** → per-conv `wa_autonomous` (миграция 017,
  `POST /conversations/{id}/wa-autonomous`): бот ведёт ИМЕННО этот диалог сам (+ сразу шлёт
  текущий черновик). 3 режима в Каналы→WhatsApp: 👀 Наблюдение / 📝 Черновик→Telegram-админу /
  🤖 Автоответ. Также починен `deliver_operator_reply` — ручной ответ из панели теперь
  реально уходит WhatsApp-лиду (раньше для WA молча «сохранялся»).

Всё за дефолтами: `wa_draft_mode=False` → поведение прода 1:1 (бот отвечает клиенту, как
раньше). Проверено: 5 юнит-тестов парсера (текст/echo/self/voice/статус) PASS; py_compile +
vite build зелёные. НЕ задеплоено на прод (по плану — сначала тест-номер).

### Как тестировать построенное (Фаза 0, тест-номер Meta)
1. Создать Meta Developer App (Business) → WhatsApp → взять тест-номер + System User token.
2. Railway → Variables: `WHATSAPP_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`, `WHATSAPP_VERIFY_TOKEN`,
   `WHATSAPP_APP_SECRET`; для голоса — `GROQ_API_KEY` (уже есть). Задать `TELEGRAM_CHAT_ID`
   (или `manager_chat_id` в настройках) — куда слать черновики.
3. Webhook → `…/webhooks/whatsapp`, подписка на поле `messages`.
4. В панели Каналы → WhatsApp включить **🛡 Режим черновика**.
5. С одного из 5 тест-номеров написать боту текст и голосовое → проверить, что в Telegram
   пришёл черновик + ТЗ с кнопками, и что ✅ реально отправляет ответ на тест-номер.

Источники research: Meta docs (coexistence, media-download, pricing 2026) — см. сессию.
